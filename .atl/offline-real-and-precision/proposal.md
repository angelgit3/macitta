# Proposal: offline-real-and-precision

## Summary
This change combines two related improvements: (1) **Offline Real** ‚Äî ensures offline sessions and reviews are properly queued in Dexie's `syncQueue` instead of silently dropped, and (2) **Precision Personal** ‚Äî computes and displays the user's precision metric (average accuracy %) in their profile by wiring up the existing `study_logs` data.

## Scope

### In Scope
- Modify `useSessionManager.startSession` / `endSession` to queue `start_session` and `end_session` ops when offline (remove `navigator.onLine` guards, generate client-side session IDs with `crypto.randomUUID()`)
- Modify `saveReview` in `studyReviewService.ts` to also write to `db.studyLogs.add()` for local offline reads (in addition to existing `syncQueue` queuing)
- Extend `useUserStats` hook with a 4th parallel query to `study_logs` and compute precision: `Math.round((sum(accuracy) / count) * 100)` or `null` if no logs
- Wire `ProfileClient.tsx` to display `stats.precision` instead of hardcoded `"‚Äî"`

### Out of Scope
- Conflict resolution for edge cases (e.g., duplicate sessions when online/offline paths cross)
- Offline read path for precision display (precision still requires network for now)
- UI for sync status or queue length display
- Per-slot or per-card precision breakdown
- Any Dexie schema migrations (all 4 tables + `syncQueue` ops already exist)

## Approach

**Offline Real**: The Dexie DB already has all infrastructure in place ‚Äî `syncQueue` supports all 5 required op types and `useSync` processes them every 30s when online. The gap is purely in the calling code: `useSessionManager` has `navigator.onLine` guards that short-circuit and return `null`/`0` without queuing anything. We remove those guards, generate session IDs client-side when offline, and queue the appropriate ops. For `saveReview`, the queueing already works ‚Äî we just add the missing `db.studyLogs.add()` call so local reads have data.

**Precision Personal**: The precision formula already exists in `MisClasesClient.tsx` (`sum(accuracy)/count * 100`). The `useUserStats` hook fetches `study_sessions`, `user_items`, and `cards` but never queries `study_logs`. We add one more parallel Supabase query, compute the metric, and add `precision: number | null` to the `UserStats` interface. `ProfileClient.tsx` then replaces the hardcoded `"‚Äî"` with the computed value.

This approach is low-risk because it reuses existing, proven infrastructure rather than introducing new systems.

## Success Criteria
1. Offline session start/end creates entries in `syncQueue` and replays on reconnect
2. Offline review creates entries in both `syncQueue` and `db.studyLogs`
3. Profile displays a numeric precision percentage when user has study logs, or `"Sin datos"` / `"‚Äî"` when they don't
4. No regression in online-only behavior

## Capabilities

### New Capabilities
- **Offline session queuing**: `start_session`, `end_session`, and `increment_session_time` ops are queued to `syncQueue` even when the device is offline, ensuring no session data is lost
- **Local study logs**: `studyLogs` Dexie table is populated during `saveReview`, enabling future offline reads of review history
- **Precision metric**: User profile displays calculated precision (average accuracy %) derived from their study logs

### Modified Capabilities
- **useSessionManager**: No longer silently drops offline session events ‚Äî now queues them for later sync
- **ProfileClient**: Replaces placeholder `"‚Äî"` with actual precision data (or fallback when unavailable)

## Rollback Plan
1. Revert the `offline-real-and-precision` branch via `git revert` or branch delete
2. No Dexie schema migration to reverse ‚Äî all tables already exist
3. No data migration to reverse ‚Äî changes only affect write/query paths, not existing data structure
4. After revert, offline sessions will again be silently dropped (pre-existing behavior); precision will return to showing `"‚Äî"`

## Risks
- **Session ID collision** ‚Äî If a user starts a session offline, goes online, and the same ID is generated by the server path, a duplicate could occur ‚Äî *Mitigation*: `crypto.randomUUID()` has astronomically low collision probability; add uniqueness constraint on `sessions.id` in Supabase as a safety net (separate follow-up task)
- **Dual-write divergence** ‚Äî `saveReview` now writes to both `db.studyLogs` (local) and `syncQueue` (for server); if one succeeds and the other fails, data could diverge ‚Äî *Mitigation*: both writes happen in the same synchronous block inside `saveReview`; if an exception occurs, neither completes. Future: wrap in Dexie transaction
- **Precision query performance** ‚Äî Adding a 4th parallel query to `useUserStats` increases load slightly ‚Äî *Mitigation*: query is `count`-based aggregation on user's own logs only (not full table scan); negligible for any realistic user data size
- **Low overall risk** ‚Äî Both changes are well-contained, reuse existing infrastructure, and have clear rollback paths

## Dependencies
- Dexie DB schema already in place (`cards`, `userItems`, `studyLogs`, `syncQueue` tables)
- `useSync` hook already processes all 5 op types every 30s when online
- No new Supabase tables, RLS policies, or API routes required
- No new npm dependencies required
